17 //================
18 // Binaryen C API
19 //
20 // The first part of the API lets you create modules and their parts.
21 //
22 // The second part of the API lets you perform operations on modules.
23 //
24 // The third part of the API lets you provide a general control-flow
25 // graph (CFG) as input.
26 //
27 // The final part of the API contains miscellaneous utilities like
28 // debugging/tracing for the API itself.
29 //
30 // ---------------
31 //
32 // Thread safety: You can create Expressions in parallel, as they do not
33 // refer to global state. BinaryenAddFunction and
34 // BinaryenAddFunctionType are also thread-safe, which means
35 // that you can create functions and their contents in multiple
36 // threads. This is important since functions are where the
37 // majority of the work is done.
38 // Other methods - creating imports, exports, etc. - are
39 // not currently thread-safe (as there is typically no need
40 // to parallelize them).
41 //
42 //================

79 // Modules
80 //
81 // Modules contain lists of functions, imports, exports, function types. The
82 // Add* methods create them on a module. The module owns them and will free their
83 // memory when the module is disposed of.
84 //
85 // Expressions are also allocated inside modules, and freed with the module. They
86 // are not created by Add* methods, since they are not added directly on the
87 // module, instead, they are arguments to other expressions (and then they are
88 // the children of that AST node), or to a function (and then they are the body
89 // of that function).
90 //
91 // A module can also contain a function table for indirect calls, a memory,
92 // and a start method.

287 // GetLocal: Note the 'type' parameter. It might seem redundant, since the
288 // local at that index must have a type. However, this API lets you
289 // build code "top-down": create a node, then its parents, and so
290 // on, and finally create the function at the end. (Note that in fact
291 // you do not mention a function when creating ExpressionRefs, only
292 // a module.) And since GetLocal is a leaf node, we need to be told
293 // its type. (Other nodes detect their type either from their
294 // type or their opcode, or failing that, their children. But
295 // GetLocal has no children, it is where a "stream" of type info
296 // begins.)
