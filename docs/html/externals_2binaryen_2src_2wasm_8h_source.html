17 //
18 // wasm.h: WebAssembly representation and processing library, in one
19 // header file.
20 //
21 // This represents WebAssembly in an AST format, with a focus on making
22 // it easy to not just inspect but also to process. For example, some
23 // things that this enables are:
24 //
25 // * Interpreting: See wasm-interpreter.h.
26 // * Optimizing: See asm2wasm.h, which performs some optimizations
27 // after code generation.
28 // * Validation: See wasm-validator.h.
29 // * Pretty-printing: See Print.cpp.
30 //
31 
32 //
33 // wasm.js internal WebAssembly representation design:
34 //
35 // * Unify where possible. Where size isn't a concern, combine
36 // classes, so binary ops and relational ops are joined. This
37 // simplifies that AST and makes traversals easier.
38 // * Optimize for size? This might justify separating if and if_else
39 // (so that if doesn't have an always-empty else; also it avoids
40 // a branch).
41 //

133 //
134 // Expressions
135 //
136 // Note that little is provided in terms of constructors for these. The rationale
137 // is that writing new Something(a, b, c, d, e) is not the clearest, and it would
138 // be better to write new Something(name=a, leftOperand=b... etc., but C++
139 // lacks named operands, so in asm2wasm etc. you will see things like
140 // auto x = new Something();
141 // x->name = a;
142 // x->leftOperand = b;
143 // ..
144 // which is less compact but less ambiguous. See wasm-builder.h for a more
145 // friendly API for building nodes.
146 //
147 // Most nodes have no need of internal allocation, and when arena-allocated
148 // they drop the provided arena on the floor. You can create random instances
149 // of those that are not in an arena without issue. However, the nodes that
150 // have internal allocation will need an allocator provided to them in order
151 // to be constructed.
